## Findings (ranked most-severe first)

- [x] **R01 — `derived()` is not slot-memoized; recomputes & reallocates every render** — `kinetica-runtime/src/ComponentScope.kt:631` — Fixed: `derived()` now backs its `DerivedCell` with a positional slot (`nextSlotKey` + `slot{}`) like `state()`, so the same instance (and its version/cache) is reused across renders instead of being reallocated each render; public signature unchanged.
  - `state()` is slot-backed (`slot(slotKey){…}`); `derived()` returns a bare `DerivedCell`. Every render mints a new
    instance (version restarts at 0), so its lazy cache/version machinery is dead across renders and memoization never happens.
  - Fix: back `derived()` with a positional slot (`nextSlotKey` + `slot{}`) like `state()`, without changing the public signature.
  - Proof: render twice reading a `derived{}`, assert the same instance/compute-count is reused; assert version survives across renders.

- [x] **R02 — `collectSuspend` uses thread-confined read-tracking; deps lost after a coroutine thread-hop** — `kinetica-runtime/src/Cell.kt:53` — Fixed: `collectSuspend` now delegates to a platform `ReadTrackingLocal.collectSuspend`; the JVM/Android actuals install a `ThreadContextElement` that pushes/pops the observer on each coroutine resume/suspend so the read-tracking frame follows the coroutine across dispatcher hops (reads after a thread-hop are tracked, origin thread's stack left clean), while single-threaded targets (js/wasmJs/macosArm64) keep the original push/try-finally/pop; synchronous per-thread `collect` isolation is unchanged.
  - `renderSuspend`/`suspendSubtree` push the observer onto a `ThreadLocal` stack, then await a `suspend` block. After a
    dispatcher hop, reads aren't tracked (subtree never re-renders) and the `finally` pop runs on the wrong thread, leaking the observer.
  - Fix: carry the read-tracking frame in a `ThreadContextElement` so it follows the coroutine; preserve the synchronous per-thread isolation.
  - Proof: `renderSuspend { withContext(Dispatchers.Default){}; readCell() }`, assert the cell became a tracked dependency and the origin thread's stack is clean.

- [x] **R03 — Reading `.value` on an unobserved derived cell subscribes it permanently (leak) + eager waste; `peek{}` doesn't prevent it** — `kinetica-runtime/src/Cell.kt:183` / `233` — Fixed: `recomputeLocked` now (re)attaches source subscriptions ONLY while the cell has ≥1 observer; unobserved reads recompute on demand without subscribing, and detect staleness lazily via a recorded per-dependency version snapshot (`dependencyVersions` + `dependenciesUnchangedLocked`) instead of an eager source listener, so a bare/`peek{}` read no longer leaks the derived cell into its sources or recomputes on every write.
  - Subscriptions are torn down only on a later recompute or when the last observer disposes; a bare read (event handler, `peek`) with no observer leaks the cell into its sources forever and recomputes on every source write.
  - Fix: subscribe to dependencies only while observed; when unobserved, recompute on-demand without registering source listeners (unify with R08).
  - Proof: `peek { derived.value }`, write source N times, assert source listener count returns to 0 and no eager recompute happens.

- [x] **R04 — `recomputeLocked` disposes all subscriptions *before* `compute()`; a throwing compute detaches the cell** — `kinetica-runtime/src/Cell.kt:172` — Fixed: reordered `recomputeLocked` to run `compute()` FIRST (collecting the new dependency set) and only reconcile subscriptions (`clearSubscriptionsLocked` + re-subscribe) AFTER compute succeeds, so a throwing compute returns early leaving existing subscriptions intact and the reactive link survives.
  - `clearSubscriptionsLocked()` runs first; if `compute()` throws, re-subscription never happens → cell left with zero subscriptions but observers registered → future dep changes silently missed.
  - Fix: collect new deps into a temp, and only swap/dispose old subscriptions after `compute()` succeeds (no teardown on the throwing path).
  - Proof: dependency-driven recompute where `compute` throws once, then a later dep change must still recompute/notify.

- [~] **R05 — Diamond dependency glitch: derived recomputes twice, publishes a transient wrong value, double-bumps version, double-notifies** — `kinetica-runtime/src/Cell.kt:184` — PARTIAL (proof passes, but a residual gap was found on audit; left DEFERRED). The incremental-reconcile + pull-based self-heal + version-dedup eliminates the *basic* diamond glitch — `DerivedCellDiamondNoGlitchTest` passes (for `C=A+B` from `S`, C recomputes and notifies exactly once with the correct value). **But** an orchestrator audit confirmed a newly-introduced missed-notification: when a shared intermediate derived `b` **self-heals during another cell's read** (e.g. `c` reads `b` while recomputing), `b`'s own dependency callback then short-circuits via `needsRecomputeLocked()`, so a *second, independent* observer `d` of `b` is **never notified** — verified with a 4-node graph: `d`'s observer fired 0 times on an `S` write; `d`'s value only corrected on a later manual read. That trades a transient self-correcting glitch for a potentially *permanent* stale observer (arguably worse). A complete fix needs a real topological/batched scheduler (mark the transitive dependent closure dirty, then flush once in dependency order, notifying each observer exactly once). Recommend implementing that scheduler or reverting R05 to accept the original transient-glitch behavior — a genuine design decision, hence DEFERRED rather than fixed.
  - `C = A+B` both from `S`: writing `S` fires A's callback → C recomputes with A(new)+B(old) (glitch), then B's callback → C recomputes correctly. No topological batching.
  - Fix: batch propagation (generation/dirty-mark + single flush) so each derived recomputes at most once per wave; suppress intermediate glitch value.
  - Proof: record every value C publishes on one `S` write; assert no intermediate `A_new+B_old` value and exactly one notification.

- [x] **R06 — `BackStack` does non-atomic read-modify-write; concurrent nav drops updates** — `kinetica-runtime/src/Navigation.kt:23` — Fixed: routed push/pop/replaceAll through the backing `MutableCellImpl.update{}` (atomic read-modify-write under the cell write lock) and overrode `BackStack.update()` to delegate atomically while re-checking the non-empty invariant inside the transform.
  - `push`/`pop`/`replaceAll` do `value = value + x` (read-then-write); inherits the racy `MutableCell.update` default. Concurrent push/pop loses an update.
  - Fix: implement push/pop/replaceAll via an atomic `cell.update{}`; keep the non-empty invariant inside the transform.
  - Proof: many concurrent `push`/`pop` from threads, assert final stack size equals net operations (no lost updates).

- [x] **R07 — Listener notification is not exception-isolated; one throwing listener aborts the rest** — `kinetica-runtime/src/Cell.kt:138` / `192` — Fixed: added a shared `notifyAll` helper that invokes every listener in its own try/catch, then rethrows the first error (with the rest as suppressed) after all have been notified; used at both notify sites (MutableCellImpl.notifyWrite and DerivedCell's dependency callback).
  - `listeners.forEach { it() }` propagates the first exception, so remaining observers are never notified (after `version` was bumped).
  - Fix: invoke each listener in a try/catch; aggregate/report errors after notifying all.
  - Proof: register a throwing listener before a good one; write; assert the good listener still fired.

- [x] **R08 — `DerivedCell.observe()` establishes no subscriptions; observe-before-first-read is inert** — `kinetica-runtime/src/Cell.kt:213` — Fixed: first observer now activates the cell (sets dirty + recomputeLocked) so dependency subscriptions exist even when `.value` was never read; re-observing after teardown re-activates.
  - `observe()` only appends a listener; a cell whose `.value` was never read has zero source subscriptions, so source changes notify nobody until a read.
  - Fix: activate dependency subscriptions on first observer (compute+subscribe); tear down on last dispose (unify with R03).
  - Proof: `derived.observe{fired=true}` (no prior read), write source, assert `fired`.

- [x] **R09 — `version++`/notify run outside `cellWriteLock` while value is committed inside; no atomic (value,version) snapshot** — `kinetica-runtime/src/Cell.kt:136` — Fixed: moved `versionCounter.incrementAndGet()` into the `synchronized(cellWriteLock)` commit block in both `update()` and `setAtomic()`, publishing the version BEFORE the value write so a lock-free reader that sees the new value always sees the matching-or-newer version; listener/onWrite dispatch stays outside the lock.
  - Reader can see the new value with a stale version; under concurrent writers, notification/`onWrite` order can diverge from commit order.
  - Fix: increment `versionCounter` inside the synchronized commit block (keep listener dispatch outside).
  - Proof: single-thread — after `value = x` returns, assert `version` already reflects the write before any listener could observe otherwise; ordering assertion for onWrite.

- [x] **R10 — Forms `FieldState` exposes validation state as plain vars, not Cells; renders never re-run on validation** — `kinetica-forms/src/Forms.kt:75` — Fixed: backed `error`/`isTouched`/`isValidating`/`hasValidated` with `store()` MutableCells read through cell-backed getters (public read shapes Boolean/String? unchanged), so reads inside a render subscribe and `set()`/`reset()`/`validate()` writes invalidate the render.
  - `error`/`isValid`/`isTouched`/`isValidating` are plain `var`s; reads record no dependency → stale validation UI after `validate()`.
  - Fix: back those fields with Cells (or a single state cell) so reads auto-subscribe and `validate()` writes invalidate.
  - Proof: render reads `field.isValid`/`field.error`; run `validate()`; assert the runtime was invalidated / dependency recorded.

- [x] **R11 — TOCTOU: derived subscribes to a dependency only *after* `compute()` read it; a concurrent write in the window is lost** — `kinetica-runtime/src/Cell.kt:182` — Fixed: `recomputeLocked` now records each dependency's version AT READ TIME (in the ReadTracking observer) and, after (re)subscribing, re-checks whether any dependency advanced during the read-to-subscribe window; if so it loops and recomputes with the fresh source values until the recorded versions are stable, so a write landing in that window is no longer lost by either the subscription snapshot or the lazy version check.
  - Between the read inside `compute()` and the `observe()` subscribe, a source write notifies a snapshot that excludes this cell → permanently stale.
  - Fix: after subscribing, re-check the dependency's version/value and recompute if it changed during the window.
  - Proof: deterministically interleave a source write between compute and subscribe (test hook), assert derived recomputes to the new value.

- [x] **R12 — Global `cellWriteLock` serializes all writes and runs user code under locks → contention + AB-BA deadlock** — `kinetica-runtime/src/Cell.kt:80` — Fixed: removed the module-global `cellWriteLock` and gave `MutableCellImpl` a per-instance `writeLock`, so writes to independent cells never contend and no single monitor can AB-BA-deadlock with `DerivedCell`'s lock; `update()`/`setAtomic()` still commit read-modify-write under the per-cell lock (atomicity preserved) and dispatch listeners/`onWrite` outside it.
  - One module-global lock for every write; `transform`/`equality`/`compute` run while locks are held, enabling cross-lock deadlock with `DerivedCell.lock`.
  - Fix: per-instance write lock; avoid holding it across user callbacks (compute equality/transform, then commit/notify without user code under the lock).
  - Proof: two independent cells written concurrently make progress; a construction that would AB-BA-deadlock under the global lock completes.

- [x] **R13 — `DerivedCell.listeners` is a `Set`; identity-equal listeners collapse, so disposing one silences the other** — `kinetica-runtime/src/Cell.kt:147` — Fixed: replaced the listener `Set` with a `List<Registration>` of per-`observe()` identity holders so identical lambdas get independent subscriptions and disposing one leaves the other live.
  - Two identical (singleton) listeners store one entry; the first `dispose()` empties the set → tears down subscriptions while the other observer is still live.
  - Fix: track subscriptions by unique token (list of registrations), so each `observe` returns an independent handle.
  - Proof: register the same lambda twice, dispose one handle, write source, assert the other still fires.

- [x] **R14 — `version` getter doesn't recompute a dirty cell (while `value` does) → version/value disagree; `SkippableNodeCache` may skip a needed re-render** — `kinetica-runtime/src/Cell.kt:154` — Fixed: `DerivedCell.version` getter now recomputes when dirty/uninitialized (under the same lock, mirroring the `value` getter) so version stays consistent with value.
  - `dependenciesUnchanged()` compares `dependency.version`; a dirty-but-unrecomputed derived returns a stale version.
  - Fix: make `version` recompute when dirty (consistent with `value`), or drive skip-detection off a mechanism that can't go stale.
  - Proof: make a derived dirty without reading it, assert `version` reflects the pending change (matches what `value` would compute).

- [x] **R15 — `KineticaRuntime` has no teardown; `renderSubscriptions`/`effectScope` leak on one-shot/server renders** — `kinetica-runtime/src/KineticaRuntime.kt:38` — Fixed: added an idempotent public `dispose()` (guarded by an atomic `disposed` flag) that disposes every `renderSubscriptions` Disposable and clears the map under `runtimeLock`, then cancels `effectScope`.
  - No `dispose()/close()` releases render subscriptions or cancels the `SupervisorJob` scope; a throwaway runtime reading a shared store leaks a listener per render.
  - Fix: add `dispose()` that disposes all `renderSubscriptions` and cancels `effectScope`.
  - Proof: render reads a shared store, `runtime.dispose()`, assert the store's listener count is 0 and the scope is cancelled.
