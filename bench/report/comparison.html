Methodology and robustness

Only metrics present in both runs are compared. No missing value is imputed. Browser identity checks cover Chromium, CPU, architecture, platform and throttle where recorded; JVM checks cover runtime and OS. Sample distributions are retained in source JSON, while this report compares medians.

Limitations

  • Percentage thresholds are practical noise guards, not confidence intervals.
  • Results from different machines, browser builds, throttle settings or JVMs are blocked by default.
  • A filtered run can compare only its common subset; aggregate changes state the number of included scenarios.
  • Bundle size comparisons are deterministic for identical build inputs; runtime measurements are not.

Recommended next steps

  • Inspect the largest same-environment regressions in the metric tables and raw samples.
  • Repeat surprising runtime changes with more iterations before accepting a baseline.
  • Promote and update documentation only from a complete all-framework run.

Further questions

Do the largest deltas reproduce in a second run, and do CPU profiles or allocation traces explain them?
